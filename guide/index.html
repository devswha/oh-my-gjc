실전 가이드

가재코드 제대로 굴리기

빈말·이론 빼고, 진짜 결과 뽑아내는 GJC 사용법. 제약 신경 끄고 일 시키는 법.

요청법 모드 선택 컨텍스트 도구 서브에이전트 검증 제한/보안 안티패턴 치트시트 환경실측 모델·프로바이더 검색엔진 최적설정 과금 우선순위 모델 선택 (증거)

1요청은 "결과"로 던져라

"방법"을 지시하면 에이전트의 판단력을 죽인다. 목표·제약·완료조건을 한 문장에 박아라.

✗ 이렇게 하지 마라

"이 파일 좀 봐줘"

"어떻게 하는 게 좋을까?"

"버그 있나 확인해줘"

✓ 이렇게 던져라

"auth.ts 토큰 만료에 race condition 있는지 찾아 고치고 테스트 추가해"

"이 함수 50ms 이하로 떨어뜨려, 벤치로 증명해"

좋은 요청의 3요소

  • 목표 — 무엇을 이뤄야 하는가
  • 제약 — 건드리면 안 되는 것, 지켜야 할 컨벤션
  • 완료조건 — 어떻게 됐을 때 끝난 건가 (테스트 통과, 벤치 수치 등)

2작업 크기에 맞는 모드

작은 버그에 ralplan 부르지 말고, 모호한 대형 기능을 "그냥 해줘"로 던지지 마라.

상황쓸 것특징
그냥 명확한 작업바로 지시스킬 안 거치고 즉시 탐색→수정→검증
요구사항이 모호함/skill:deep-interview소크라테스식 질문으로 스펙부터 추출
요구 명확, 설계가 위험/skill:ralplanPlanner·Architect·Critic 합의 플랜. 승인 전 코드 안 건드림
긴 다중 목표 추적/skill:ultragoal목표 상태·체크포인트·증거 장부
워커 병렬 실행/skill:teamtmux 기반 다중 워커 조율
핵심 흐름 모호함 → deep-interview(스펙) → ralplan(합의 플랜) → 승인 → 실행. 계획 단계는 승인 전까지 제품 코드를 절대 수정하지 않는다.

3컨텍스트를 미리 다 줘라

병렬 서브에이전트는 네 대화 히스토리를 못 본다. "전에 말한 그거"는 안 통한다.

  • 파일 경로를 정확히 박아라 — 추측하게 만들지 마라
  • 지켜야 할 컨벤션·패턴을 처음에 명시
  • 큰 자료는 붙여넣기보다 파일/URL 경로로 넘겨라
  • 하지 말아야 할 것(non-goal)도 적어라 — 범위 폭주 방지
# 좋은 위임 한 방
src/payment/refund.ts 의 processRefund 가 부분환불 시
잔액을 음수로 떨어뜨림. 음수 가드 추가하고,
기존 패턴(Result<T,E> 반환) 따라서, refund.test.ts 에
음수·0·초과 케이스 테스트 추가. UI 는 건드리지 마.

4도구를 믿어라

셸 cat/grep/sed로 일일이 시키지 마라. 전용 도구가 더 빠르고 정확하다.

read 읽기

파일·디렉토리·아카이브·PDF·이미지·SQLite·URL 전부 한 도구로. 섹션 단위로 읽음.

search 내용 검색

정규식 기반. grep/rg 대체. .gitignore 인식.

find 파일 찾기

glob 기반 파일명 탐색. 최근 수정순 정렬.

lsp 코드 인텔

정의·참조·구현·타입·리네임. "이 심볼 어디 쓰여?" → 정확히 추적.

ast_grep / ast_edit 구조

구문 단위 검색·코드모드. 텍스트 치환보다 안전.

debug 디버거

DAP 기반. 브레이크포인트·스텝·변수 검사·멈춘 프로세스 인터럽트.

5큰 작업은 쪼개서 병렬로

다중 파일 리팩터·독립 모듈 작업은 서브에이전트로 fan-out. 단, 의존 없을 때만 이득.

executor

경계가 명확한 구현·수정·리팩터. 실제 코드 편집 담당.

planner

읽기 전용. 순서·완료조건·리스크 매핑.

architect

읽기 전용. 아키텍처 평가·코드 리뷰(심각도 등급).

critic

읽기 전용. 플랜 비판. 추측 없이 실행 가능할 때만 승인.

병렬의 철칙 "B가 A의 출력을 봐야 하나?" → 그렇다면 순차. 디스크상 파일이 안 겹치고 독립적일 때만 병렬이 진짜 이득이다.

6검증을 강제해라

"고쳤다"는 말 믿지 말고 증거를 요구해라.

✓ 증거 요구 문구 "테스트 돌려서 통과 출력 보여줘" · "그 시나리오 실제로 재현해봐" · "벤치 수치 before/after 보여줘"
  • 관찰 가능한 동작·엣지값·분기 조건·에러 처리를 테스트
  • 기본값이나 동어반복(tautology)은 테스트하지 마라 — 의미 없다
  • 비자명한 변경은 검증 없이 끝내지 않는 게 기본 동작

7"제한 없이"에 대한 솔직한 답

코드 작업 자체엔 사실상 제약 없다.

시스템 깊은 곳 디버깅, 바이너리 분석·리버싱, 익스플로잇 PoC, 네트워크 도구, 저수준 메모리 조작 — 뭐든 짠다. 막히는 거의 유일한 선은 단 두 개:

차단되는 것

남의 작업 날리기 — 명시적 지시 없이 revert / stash / force-push / 삭제 안 함. (검열이 아니라 안전장치)

거짓 보고 — 안 돌린 테스트를 돌렸다고 하지 않음. 출력 날조 안 함.

팁

보안·저수준 작업이면 목적을 명확히 말해라. 우회하느라 시간 버리지 않게.

이 두 선만 빼면 "보안 상관없이" 굴려도 된다.

8안티패턴 모음

하지 마라대신
"전체 코드베이스 분석해줘"구체적 목표/파일을 줘라. 막연한 분석은 토큰만 태움
작은 수정에 ralplan/team 호출그냥 바로 지시

"대충 알아서"로 대형 기능 요청deep-interview로 스펙부터

셸 grep/sed/cat 떡칠search/edit/read 도구

11검색 엔진 설정 완료 · 검증됨

처음엔 유료 검색 키가 0개라 auto가 DuckDuckGo로 폴백했다. 지금은 Exa 키 적용 + 작동 검증 완료(Provider: Exa 확인).

핵심 함정 (꼭 기억) gjc 는 프로젝트(cwd) .env 의 키를 보안상 의도적으로 무시한다. 신뢰 위치에서만 읽음 — 우선순위 라이브env → ~/.gjc/agent/.env → ~/.gjc/.env → ~/.env. 그래서 레포 .env 를 ~/.gjc/.env 로 심링크해 둠. source 불필요.

결론 (앞 추천 정정) 실제 코드 작성은 Codex가 분명히 위. 너는 Codex Pro(정액)라 executor를 Codex로 돌려도 돈 추가 없음 — 쿼터만 씀. 그래서 executor→Kimi 자동 라우팅은 "품질 깎아 쿼터 절약"이라 쿼터가 실제 병목일 때만 이득. → 기본은 executor도 Codex 유지(현 task.agentModelOverrides={} 그대로). Kimi는 대량 병렬·툴헤비 작업에 gjc --mpreset kimi 수동 레인.

  • lsp.formatOnWrite = false — 프로젝트 포매터 충돌 방지(임의 리포맷 금지 원칙)
  • temperature = -1 — 프로바이더 기본. 코딩엔 손대지 마라
  • modelRoles default = gpt-5.5:xhigh — 코딩 최적. 더 빡세게는 claude-max 프로필

A+B 한 방 적용

gjc config set bashInterceptor.enabled true
gjc config set checkpoint.enabled true
gjc config set bash.autoBackground.enabled true
gjc config set lsp.diagnosticsOnEdit true
gjc config set task.maxConcurrency 12
gjc config set task.enableLsp true
gjc config set async.enabled true
되돌리기 각 키는 gjc config reset <키> 또는 set ... false로 원복. 안전하게 실험 가능.
