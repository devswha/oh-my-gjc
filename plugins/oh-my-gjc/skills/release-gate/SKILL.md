# release-gate — 3게이트 릴리스 절차

목적: "에이전트가 검토 없이 혼자 main에 머지+태그" 사고(oh-my-gjc 2026-07-08, self-merge
4연발)를 구조적으로 차단하는 릴리스 세리머니. 각 게이트는 서로 다른 실패 모드를 막는다 —
**기계가 깨짐을 잡고(G1), 맥락 없는 남이 자기합리화를 잡고(G2), 사람이 책임지고(G3),
전 과정이 증거로 남는다.** 실증: oh-my-gjc v0.15.1·v0.16.0에서 리뷰어가 릴리스 3연속으로
실블로커(수치 과장·표면 누락·기본값 파손 풋건)를 잡아냄.

## Gate 1 — 검증 (기계적, 로컬)

레포에 존재하는 검증 수단을 전부 실행하고 **증거(rc·카운트)를 기록**한다:
- 정적: 매니페스트/설정 파스(JSON/YAML), 버전 필드 일치, `bash -n` 등 문법 검사.
- 단위/통합 테스트: 프로젝트 러너(`bun test` 등) — 실패 0이 전제.
- 재현: 신규 설치/빌드를 **격리 환경**(빈 HOME, 클린 클론)에서 끝까지 — "내 머신에선 됐는데" 차단.
- 검증 불가 표면은 pending-environment로 명시(가짜 증거 금지).

## Gate 2 — 외부 교차리뷰 (fail-closed)

작업자가 자기 결과에 도장 찍는 것을 차단한다. `extragoal` 스킬의 리뷰 레인과 동일 기계를
쓴다(재구현 금지):

1. **클린 트리 확정:** 버전 범프 등 릴리스에 들어갈 모든 변경을 먼저 커밋하고
   `git status --porcelain`이 비어 있는지 확인한다. **서명 대상 = 그 HEAD 커밋으로 고정** —
   서명 후 커밋이 하나라도 추가되면 그 서명은 무효, 새 HEAD로 재서명한다.
2. 릴리스 디프(`git diff <직전태그>..HEAD`) 생성 → **이그레스 전 시크릿 스캔**(발견 시 중단).
3. **fresh-context 리뷰어** 기동: `GJC_NOTIFICATIONS=0 gjc -p --no-session --model <셀렉터> --tools read,search,find` — 세션 무저장, 읽기 3도구만(`goal` 등 상태 도구 미노출), 작업 맥락 미공유. 리뷰어는 **가능하면 저작 모델과 교차 패밀리**로 — 불가하면 사유를 증거 문서에 기록.
4. 디프 + 레포 읽기 권한 + 구체 점검 지시를 주고, 마지막 줄에 정확히
   `VERDICT: APPROVE` 또는 `VERDICT: REQUEST_CHANGES`만 출력하게 한다.
5. **Fail-closed:** verdict 부재/기형/타임아웃/REQUEST_CHANGES ⇒ 발행 불가.
   지적 수정 → 커밋 → **새 fresh 리뷰어로 새 HEAD에 재서명, 최대 2라운드.** 한도 소진 시
   릴리스를 멈추고 사람에게 보고한다(억지 통과 금지).

## Gate 3 — 인간 승인 (자기 승인 절대 금지 — 약화 불가 계약)

게이트 1·2 증거와 **최종 후보 커밋 해시**를 첨부해 운영자 승인을 요청하고 멈춘다
(관제탑 큐 또는 세션 질의). **에이전트가 스스로 승인하는 경로는 없고, 작업 시작 전의
"릴리스해라" 지시는 게이트 착수 권한이지 발행 승인이 아니다** — 승인은 반드시
G1·G2를 통과한 최종 후보를 보고 이뤄져야 한다(사전 지시가 있으면 관제탑 큐 대신
세션에서 바로 물어도 된다는 의미일 뿐이다).

## 통과 후 — 기계적 발행 + 증거 박제

- **발행 대상 = 승인된 HEAD의 트리, 정확히 그것.** 발행 경로는 레포 거버넌스를 따른다:
  branch-flow/브랜치 보호가 있는 레포는 release PR(`gh pr create --base main --head dev`)
  + CI 통과 후 머지, 없는 레포는 로컬 `dev`→`main` `--no-ff` 머지 → 주석 태그 → 릴리스
  발행(GitHub Release 등). 머지 커밋은 승인 HEAD만 담아야 하며 **트리가 승인 HEAD와
  동일함을 확인**한다(`git diff <승인해시>..main --stat`이 비어야 함).
- **증거 문서는 발행 후에 커밋한다** (r2 finding 4 — 발행 전에 커밋하면 실제 발행
  HEAD가 미승인 커밋이 되는 자기모순): `docs/verification/<프로젝트>-release-vX.Y.Z-<날짜>.md`를
  발행 직후 dev에 **docs-only 커밋**으로 남긴다. 릴리스 태그에는 포함되지 않고(코드가
  아닌 기록이므로 승인 무효화 없음) 다음 릴리스 디프 범위에 자연 포함된다.
  내용: 게이트 1 체크리스트 결과, 게이트 2 전 라운드 verdict·블로커·수정 커밋,
  게이트 3 승인 근거(승인된 해시 포함).
- 빈도 캡(기본: 문서·패치 번들 1일 1릴리스). **예외는 긴급 security/install-breakage
  수정뿐**이며 그때도 게이트 1·2는 그대로 실행한다.
- 롤백은 **fix-forward만**: revert 커밋 → 게이트 1·2 재실행 → 새 패치 릴리스.
  태그/릴리스 삭제·강제 이동 금지(superseded 표기).

## 절대 규칙 (약화 금지)

- 자기 승인 금지 · verdict fail-closed · 재서명 ≤2라운드 · 시크릿 스캔 선행 ·
  증거 없이 발행 금지 · 부분 통과("G1만 통과했으니 일단") 발행 금지.
- 관측 증거는 실제 실행 결과만 기록한다 — 검증 주장과 실행 내역이 일치해야 한다.
