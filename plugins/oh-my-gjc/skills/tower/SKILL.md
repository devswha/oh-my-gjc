# tower — 관제탑 (상주 세션 오케스트레이션)

여러 TUI 에이전트 세션(기본: gjc, tmux 창 이름 접두어 `GJC-`)을 **하나의 관제탑
세션**에서 감시·전파·중재한다. 관제탑은 방향/우선순위/컨텍스트를 각 세션에 주입하고
상태를 글랜스하며, 사람이 결정해야 할 항목을 큐에 모아 보고한다. 세부·위험 결정은
사람이 해당 세션에 직접 붙어서 한다.

원본 부품은 이 플러그인의 `bin/`에 있다(무의존 stdlib/bash):
- `session_watch.py` — busy→idle 완료 감시기(이벤트 소스).
- `tower-notify.sh` — tmux 세션 메시지 주입 + TUI 함정 방어.
- `queue_store.py` + `tower` CLI — 운영자 결정 큐.

## 설정

`bin/` 경로 해석(설치 캐시): `~/.gjc/plugins/cache/plugins/oh-my-gjc___oh-my-gjc___*/bin/` (`sort -V | tail -1`로 최신). 프로젝트/레포
설치면 각 스코프의 폴백. 값은 **설정 파일 또는 환경변수**로 준다(하드코딩 금지):
- `TOWER_WINDOW_PREFIX`(기본 `GJC-`) — 감시 대상 세션의 tmux 창 이름 접두어.
- `TOWER_QUEUE_FILE` > `TOWER_HOME/queue.json` > `~/.gjc/tower/queue.json` — 큐 경로.
- `session_watch.py --config <json>` — window_prefix/busy_mark/idle_mark/exclude/interval/confirm.
  예시: `references/tower.config.example.json`.
- 관제탑 세션 자신은 반드시 감시 제외(`--exclude <자기세션>` 또는 config `exclude`).

## 운영 루프 (이 순서·규약을 지킨다)

### 1) 부팅 재등록 (세션 귀속 — 필수)
감시·순찰은 **세션에 귀속**된다. 관제탑 세션을 새로 열/재개할 때마다 재등록한다:
- **실시간 감시**: `monitor` 도구로 `session_watch.py`를 상주 실행
  (`bun`/`python3 <bin>/session_watch.py --exclude <자기세션> [--config …]`). 이미
  떠 있으면 중복 실행 금지(프로세스 확인 후).
- **백업 순찰**: 30분 주기 자동 순찰을 예약(gjc cron 예약 프롬프트). 실시간 감시가
  놓친 것을 수확하고 알림 단일화 정책을 집행하는 백업 스윕.
- 이 SKILL 문단이 곧 재등록 지침이다 — 재개 시 여기를 따른다.

### 2) 이벤트 수신 시 (실시간)
`session_watch.py`가 `IDLE <세션> | <마지막 줄>` 또는 `INIT <세션> <상태>`를 방출하면:
- ①**판별** — 해당 세션 `tower peek`로 완료/결정대기/질문을 구분.
- ②**적재** — 사람 처리가 필요하면 `tower queue add <세션> "<요약>"`(멱등 — 동일
  출처+텍스트 대기 중복 방지).
- ③**보고** — 사람에게 즉시 한 줄 요약 보고(완료든 블록이든). 알림은 단일 채널로.
- ④**릴레이** — 기결정의 후속이면 결정 루프로 해당 세션에 전파(아래).

### 3) 순찰 (백업, 30분)
글랜스(`tower peek`) → 새 완료 감지 → 큐 적재 → **새 항목만** 보고.
**빈 순찰은 무보고**(변화 없으면 아무 알림도 내지 않는다).

### 4) 결정 루프
사람 결정 발생 → ①관련 기록 갱신 → ②관련 세션에 `tower notify`로 전파 →
③다음 병목을 사람에게 보고 → 처리된 큐 항목은 `tower queue done <id>`.

### 5) 알림 단일화
원격 알림/조종은 **관제탑 세션 하나로만**. 다른 세션 직통 알림은 관제탑 경유로 모은다.
(새 세션이 전역 설정 때문에 알림에 자동으로 다시 붙을 수 있으니, 글랜스 때 정책 이탈을
발견하면 해당 세션에 알림 off를 주입해 유지.)

## TUI 주입 함정 (방어 로직 + 규칙)

tmux `send-keys`로 TUI 입력창에 텍스트를 넣을 때 세 가지 함정이 있다. `tower-notify.sh`가
방어하지만, 메시지를 만들 때 규칙도 지킨다(2026-07 실측):

1. **물결 `~`** — TUI 경로 자동완성이 입력을 오염시킨다. → 스크립트가 **거부**. 메시지에
   `~` 쓰지 말 것("약" 등으로 대체).
2. **괄호+대문자 `(X)`** — TUI가 이모지로 치환한다. → 스크립트가 **거부**. 넘버링은
   `①②③`을 쓸 것.
3. **실존 경로 토큰** — 대상 세션 cwd에 실재하는 상대경로(`docs/FOO.md` 등)가 메시지에
   들어가면 자동완성 팝업이 Enter를 삼켜 **미전송+본문 오염**이 된다. URL(`://`)은 무관.
   → 스크립트가 **경고**하고, 전송 후 입력창 잔류를 검증해 Enter 1회 재시도한다. 그래도
   남으면 `FAIL`을 보고(수동 확인). **규칙: 파일 지칭은 말로 풀어 쓸 것**("assets 아래
   demo 폴더의 X 파일"). 미전송 의심 시 대상 pane 하단 입력창을 Escape·C-u로 비우고 재전송.

## 모델 인지 전파 (선택)
세션마다 실행 모델이 다르면(`tower models` — gjc 세션 로그 최빈 모델) 그 모델 특성에 맞게
프롬프트를 조정한다. 특히 보안/적대적 프레이밍에 민감한 모델에는 방어자·코드리뷰
프레이밍을 쓰고, 관제탑 세션 자체는 그런 내용을 릴레이·요약해야 하므로 그런 필터가 있는
모델을 관제탑에 두지 않는다.

## 경계 (안전)
관제탑은 방향·컨텍스트 주입, 상태 확인, (사람이 명시 지시하고 하류에 안전망이 있을 때)
승인 게이트 대리 조작까지 한다. **항상 사람 본인만**: 실거래 주문·청산 확정, 실결제·자금
이동, 라이브 발행 스위치, 금융 계정 로그인/조작. 결정 큐는 "사람이 처리할 것"을 모을 뿐,
관제탑이 그 결정을 대신 내리지 않는다.

## gjc `team`과의 차이 (명시)
- **team** = 하나의 작업을 여러 **작업 워커**에게 나눠 조율하는 실행 오케스트레이션
  (공유 상태·메일박스·워크트리·검증 레인). 워커는 과제를 받아 코드를 만든다.
- **tower** = 이미 각자 돌고 있는 **상주 함대**(독립 세션들)를 감시하고, 방향을 주입하고,
  **사람이 내릴 결정**을 큐에 모아 보고하는 상주 감독. 관제탑은 일을 나눠주는 게 아니라
  상태를 관측하고 사람 결정을 중개한다.
- 요약: team은 *작업 분배·조율*, tower는 *상주 관측 + 사람 결정 큐*. 겹치면 team으로
  워커를 돌리고 tower로 그 세션들을 감시하는 조합도 가능하다.
