# Preset Fit (작업 크기 측정 → 빌트인 프리셋 추천)

목적: "이 레포는 모노레포인가?"라는 라벨 논쟁 대신 **이 작업의 심사 표면(working set)이
몇 토큰인가**를 repomix로 실측하고, 그 숫자를 컨텍스트 예산 밴드에 대조해 **GJC 빌트인
프리셋**과 재기동 커맨드 한 줄을 돌려준다.

커스텀 프리셋 사본을 만들거나 배포하지 않는다(은퇴한 `multivendor-presets`의 교훈 —
로컬 사본은 썩는다). 추천 대상은 항상 `gjc` 빌트인 프로파일이다.

## 전제 조건

- Node + `npx` (repomix는 `npx -y repomix@1.15.0`로 실행 — 이 레포에서 검증된 핀).
- `npx`가 없으면 측정 없이 중단하고 설치 안내만 한다(추측 밴드를 출력하지 않는다 — fail-closed).

## 절차

### 1. 스코프 결정

- 인자로 글롭이 오면 그대로 쓴다.
- 태스크 설명이 오면 read/search/find로 **관련 파일 집합**을 구성한다:
  구현 대상 파일 + 직접 콜사이트 + 해당 테스트. 레포 전체가 아니라 작업이 물어야
  하는 집합만. 스코프가 비면 측정하지 말고 스코프를 먼저 물어라.
- `--repo`면 레포 전체를 잰다(레포 급 감각용 — 판정 기준은 여전히 작업 집합임을 명시).

### 2. 측정

```sh
tmp="$(mktemp -d)"
npx -y repomix@1.15.0 --include "<glob1>,<glob2>,…" \
  --token-count-encoding o200k_base -o "$tmp/pack.xml" . \
  | tee "$tmp/summary.txt"
# "Total Tokens" 라인에서 토큰 수를 파싱한 뒤 즉시 정리:
rm -rf "$tmp"
```

- 산출물은 임시 디렉터리에만 쓴다. 레포·`.gjc/`에 아무것도 쓰지 않는다.
- pack 파일은 파싱 직후 삭제하며 어디에도 전송하지 않는다(로컬 측정 전용).
- repomix 기본 보안 검사(secretlint)는 끄지 않는다.
- `npx`는 패키지 레지스트리에 접근할 수 있다(네트워크 정책 적용)를 고지한다.

### 3. 판정 밴드 (o200k_base, 작업 집합 기준)

근거: openai-codex 계열 컨텍스트 캡 ~272k − 시스템 프롬프트·대화 누적·출력 여유
= 실효 ~200k. 리뷰 좌석(architect)은 forkContext로 입력이 밀려오는 좌석이라 캡에
가장 민감하고, planner/executor는 스스로 캐거나 조각만 받아 여유가 크다.

| 측정 토큰 | 판정 | 추천 빌트인 |
|---|---|---|
| < 60k | 어떤 빌트인이든 무사 | `opus-codex` (품질 기본) — 저지연이 더 급하면 codex 티어도 가능 |
| 60k–150k | 전-codex 티어(`codex-*`)의 리뷰 좌석 여유 부족 | `opus-codex` (리뷰 좌석이 Claude라 캡 회피) |
| > 150k | codex 좌석 배제 영역 | `claude-opus` (전 좌석 실효 장문) |

- 400k↑ 단일 입력은 프리셋과 무관하게 한 메시지로 못 넣는다(청크 필요)를 함께 경고한다.
- 밴드 수치는 2026-07 카탈로그 기준이며 모델 캡이 바뀌면 갱신 대상임을 명시한다.

### 4. 보고

다음 형식으로 보고하고 끝낸다(추가 행동 없음):

- 측정: 파일 N개, ~X 토큰 (o200k_base)
- 판정: 밴드와 근거 한 줄
- 추천: `gjc --mpreset <name>` — 재기동 커맨드 그대로. 영구 적용은 `--default` 옵션 안내.
- **모델 프로파일은 세션 중 전환 불가**(기동 시 결정)를 반드시 명시한다.

## 하지 않는 것

- `models.yml`·`config.yml`·프리셋 정의 수정, 프로파일 생성·병합.
- 측정 pack의 보존·전송, 레포 내부에 파일 생성.
- 추측 기반 추천(측정 실패 시 밴드 판정을 내리지 않는다).
- 세션 내 모델 전환 시도.
