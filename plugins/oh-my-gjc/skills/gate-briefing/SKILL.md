# Gate Briefing (승인 게이트 브리핑)

목적: 승인 게이트의 목적은 사용자의 **통과**가 아니라 **이해**다. 도메인 지식이 없는
사용자도 "내가 지금 무엇을 승인하는지"를 판정할 수 있게, 계획 원문을 근거로 브리핑을 만든다.

## 언제 활성화하나

- ralplan이 pending-approval 상태로 멈췄을 때 (`.gjc/plans/ralplan/**/pending-approval.md`,
  `.gjc/_session-*/plans/ralplan/*/pending-approval.md`)
- ultragoal/워크플로가 사용자 승인을 기다릴 때
- 사용자가 계획·게이트에 대해 "이해 안 된다", "승인해도 되냐", "무슨 뜻이냐"고 물을 때

## 브리핑 생성 절차

1. **원문을 읽는다.** pending-approval.md(또는 해당 게이트 산출물)와 마지막
   critic/architect 스테이지 파일을 실제로 읽는다. 기억이나 요약본으로 대체 금지.
2. **아래 4부 형식으로 출력한다.**

### ① 비전문가 번역
계획이 하려는 일을 일상어로. 도메인 용어는 처음 나올 때 괄호로 뜻 병기.
"무엇을, 왜, 어떤 순서로"가 그림으로 그려지게. 5문장 이내.

### ② 승인의 경계
- **지금 승인하는 것**: 이 승인으로 즉시 일어나는 일.
- **승인하지 않는 것**: 이 승인으로도 일어나지 않는 일 (특히: 실환경/실서비스/실계좌/
  프로덕션 접촉이 시작되는가? 이후 단계에 별도 승인 게이트가 더 있는가?).

### ③ 도메인-무지 체크리스트 (원문 근거 필수)
| 질문 | 답 + 근거 위치 |
|---|---|
| 잘못되면 되돌릴 수 있나? (롤백 경로) | |
| 이 승인으로 실환경이 바뀌나? 바뀐다면 언제·어떤 추가 게이트 후에? | |
| 성공/실패를 도메인 지식 없이 관측할 수 있는 지표가 정의돼 있나? | |
| 위험한 수치/임계값을 지금 확정하나, 증거(테스트·백테스트·실측)로 도출하나? | |
| 기존 안전장치를 약화·제거하는 항목이 있나? | |
| 합의 게이트 판정은? (critic verdict / architect status 원문) | |

각 행은 계획 원문에서 확인한 것만 적는다. **원문에 없으면 지어내지 말고
"명시 없음 — 승인 전 확인 요망"으로 표기한다.** 이 표기 자체가 판단 재료다.

### ④ 판정
- 추천: 승인 / 보류(확인 질문 목록 첨부) / 반려(이유).
- 승인 전에 세션에 물어볼 가치가 있는 질문이 있으면 그대로 복사해 쓸 수 있는
  형태로 1~3개 제시.

## 규칙

- **정확성 > 쉬움.** 쉽게 바꾸다 의미가 틀어질 부분은 전문용어를 유지하고 뜻을 병기.
- **승인 대행 금지.** 판정은 추천까지만. 승인/반려 실행은 사용자의 명시적 지시로만.
- 체크리스트에서 "명시 없음" 행이 2개 이상이면 추천은 자동으로 "보류"다.
- 위험·주의·사용자가 직접 해야 할 행동은 절대 생략하지 않는다.
- 군더더기 금지. 브리핑은 화면 한 장 분량을 목표로 한다.
